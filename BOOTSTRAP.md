# LLM Wiki — полная методика

Подробное описание концепции. Используется как справочник.  
Оперативные правила для Claude — в `SKILL.md`.

---

## Концепция в одном абзаце

LLM Wiki — персональная база знаний из markdown-файлов, которую пишет и поддерживает LLM, а не человек. Владелец курирует raw-источники и задаёт вопросы; LLM компилирует, обновляет, поддерживает кросс-ссылки. Wiki — компилируемый артефакт, накапливается со временем. Это **не RAG**: знание не извлекается заново на каждый запрос, а живёт постоянно в .md-страницах.

Источник паттерна: [Karpathy LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f).

---

## Три слоя

### raw/ — Immutable sources
Неизменяемая коллекция источников. LLM только читает.  
Сюда складываются: голосовые транскрипты, статьи, PDF, скриншоты, выгрузки из других систем.  
Папки: `voice/`, `articles/`, `screenshots/`, `obsidian/`.

### llm-wiki/ — LLM-exclusive write
Markdown-страницы по сущностям, концепциям, задачам, идеям.  
LLM пишет и обновляет. Владелец редактирует редко и осознанно.  
Содержит `CLAUDE.md` с **только локальной** спецификой проекта.

### llm-wiki/templates/ — shared
Шаблоны для новых страниц. Часть llm-wiki, не отдельный слой.

---

## Три операции

### Ingest
Новый файл появился в `raw/` → LLM читает → при неочевидной сути обсуждает 3–5 ключевых takeaways → пишет summary-страницу + обновляет до 15 связанных страниц → обновляет `index.md` → пишет в `log.md` → обновляет `raw_read_last_date`.

### Query
Владелец задаёт вопрос → LLM читает `index.md` → открывает 3–7 релевантных страниц → синтез с цитатами. Хороший ответ сохраняется как новая wiki-страница (`source: query-derived`).

### Lint
Периодический health-check: противоречия, stale claims, orphan-страницы, отсутствие кросс-ссылок, страницы без frontmatter. Findings выводятся владельцу. Auto-fix структурных вещей — только после подтверждения.

## Routing в деталях

По умолчанию (нет подпроектов): `llm-wiki/<тип>/<slug>.md`

При наличии подпроекта: `llm-wiki/projects/<slug>/<тип>/<slug>.md`

Подпроект создаётся явно по команде владельца и имеет **голографическую структуру** — внутри `projects/<slug>/` находится полная копия: свой `CLAUDE.md`, свои папки по типам. Это позволяет каждому подпроекту развиваться независимо, используя ту же методику.

---

## Принцип разделения труда

> Владелец: читает, анализирует, задаёт вопросы, курирует источники.  
> LLM: обновляет кросс-ссылки, поддерживает согласованность, компилирует знание, следит за противоречиями.

Обслуживание дёшево, потому что LLM не устаёт от рутины.

---

## Per-project CLAUDE.md — только локальное

`llm-wiki/CLAUDE.md` в каждом проекте содержит **только**:
- Список project slugs в taxonomy
- Специальные правила этого проекта (если есть)

Общие правила (routing, frontmatter, операции) — в скилле. Не дублировать.

---

## Масштабирование

- До ~100 страниц: `index.md` достаточно для навигации.
- При росте: добавить полнотекстовый поиск (fzf, Obsidian Search, ripgrep).
- Git обеспечивает историю версий.
- Obsidian — рекомендуемый viewer: graph view, Dataview для динамических таблиц, Web Clipper для импорта.
